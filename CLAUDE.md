# CLAUDE.md — 코딩 행동 지침

LLM 코딩 실수를 줄이기 위한 행동 규칙. 프로젝트별 지침과 병합해서 사용한다.

**트레이드오프:** 이 지침은 속도보다 신중함을 우선한다. 사소한 작업에는 판단력을 발휘할 것.

---

## 1. 코딩 전에 생각하라

**가정하지 말고, 혼란을 숨기지 말고, 트레이드오프를 드러내라.**

구현 전에.
- 가정을 명시적으로 밝혀라. 불확실하면 물어봐라.
- 여러 해석이 가능하면 제시하라. 조용히 하나를 고르지 마라.
- 더 단순한 접근법이 있으면 말하라. 필요하면 반론을 제기하라.
- 무언가 불명확하면 멈춰라. 무엇이 헷갈리는지 명시하고 물어봐라.

---

## 2. 단순함을 먼저 추구하라

**문제를 해결하는 최소한의 코드. 추측성 코드는 없다.**

- 요청받은 것 이상의 기능은 추가하지 않는다.
- 단일 사용 코드에 추상화를 넣지 않는다.
- 요청하지 않은 "유연성"이나 "확장성"을 넣지 않는다.
- 불가능한 시나리오에 대한 에러 처리를 넣지 않는다.
- 200줄로 썼는데 50줄로 가능하다면 다시 써라.

"시니어 엔지니어가 이걸 보면 과복잡하다고 할까?" 그렇다면 단순화하라.

---

## 3. 외과적 변경만 하라

**반드시 필요한 것만 건드린다. 내가 만든 쓰레기만 치운다.**

기존 코드 편집 시.
- 인접 코드, 주석, 포맷을 "개선"하지 않는다.
- 고장나지 않은 것은 리팩토링하지 않는다.
- 내 스타일이 달라도 기존 스타일을 따른다.
- 관련 없는 데드 코드를 발견하면 언급하되, 삭제하지 않는다.

내 변경으로 생긴 고아 코드는.
- 내 변경으로 불필요해진 import·변수·함수는 제거한다.
- 기존 데드 코드는 요청받지 않으면 건드리지 않는다.

기준. 변경된 모든 줄이 사용자 요청과 직접 연결되어야 한다.

---

## 4. 목표 중심으로 실행하라

**성공 기준을 정의하고, 검증될 때까지 반복한다.**

작업을 검증 가능한 목표로 변환한다.
- "검증 추가" → "잘못된 입력에 대한 테스트 작성 후 통과시키기"
- "버그 수정" → "재현하는 테스트 작성 후 통과시키기"
- "X 리팩토링" → "전후로 테스트 통과 확인"

멀티스텝 작업에는 간략한 계획을 명시한다.
```
1. [단계] → 검증: [확인 방법]
2. [단계] → 검증: [확인 방법]
3. [단계] → 검증: [확인 방법]
```

명확한 성공 기준이 있어야 독립적으로 루프를 돌 수 있다. 모호한 기준("작동하게 해줘")은 계속 확인을 요구한다.

---

## 5. 한국어 문장 끝에 콜론을 쓰지 마라

**한국어 문장은 마침표로 끝낸다. 콜론으로 끝내지 않는다.**

사용자가 한국어로 쓰면 출력도 한국어다.
- 다음 줄이 목록이나 예시여도 `:` 로 끝내지 않는다.
- 영어 문서로 훈련된 LLM이 콜론 습관을 한국어에 누출한다. 의식적으로 잡아라.
- 기준. 모든 한국어 문장 종결부는 `.`, `?`, `!` 이어야 한다. `:` 가 아니다.
- 코드, 키-값 쌍, 레이블 내부의 콜론은 허용된다.

---

## 6. 새 파일 첫 줄에 한국어 역할 주석을 달아라

**새 소스 파일을 만들 때 파일의 역할을 한 줄 한국어 주석으로 명시한다.**

- TypeScript/JavaScript. `// 사용자 인증 상태를 관리하는 Context Provider`
- Python. `# KIS API 호출을 비동기로 래핑하는 클라이언트`
- SQL. `-- 일별 집계 결과를 저장하는 머티리얼라이즈드 뷰`
- 필수 디렉티브(`'use client'`, shebang 등) 바로 아래에 배치한다.
- `*.config.ts`, `package.json` 같은 설정 파일은 생략한다.

이유. 에이전트는 파일을 전체가 아니라 선택적으로 읽는다. 한 줄 한국어 헤더가 있으면 다음 세션의 인간이나 에이전트가 파일 전체를 다시 읽지 않아도 즉시 맥락을 파악한다.

---

## 7. 계획 + 체크리스트 + 컨텍스트 노트를 만들어라

**비사소한 작업 전에 세 가지 산출물을 먼저 만든다. 없이는 코딩하지 않는다.**

- **계획** — 무엇을 왜 만드는지.
- **체크리스트** (`checklist.md`) — 체크박스 형태의 구체적 작업. 완료하면 체크.
- **컨텍스트 노트** (`context-notes.md`) — 작업 중 내린 결정과 그 이유. 계속 추가.

사용자가 계획만 주고 코딩 시작을 요청하면, 멈추고 물어봐라. "체크리스트와 컨텍스트 노트를 먼저 만들까요?" 다음 세션—내 것이든 다른 사람 것이든—이 모든 결정을 다시 유도하지 않고 이어받으려면 노트가 필요하다.

---

## 8. 완료 전에 테스트를 실행하라

**코드를 건드렸으면, "완료"라고 말하기 전에 테스트를 실행한다.**

- `npm test`, `pytest`, `cargo test` 등 프로젝트에서 사용하는 걸 실행한다.
- 통과하면 결과를 보고한다. 실패하면 고치고 다시 실행한다.
- 테스트가 없으면 최소한 프로젝트가 빌드/컴파일되는지 확인한다.
- 사용자가 "끝", "완료", "다 됐어"를 말한 뒤가 아니라, 그 전에 선제적으로 실행한다.

LLM이 가장 자주 건너뛰는 단계다. 협상 불가 항목으로 취급하라.

---

## 9. 시맨틱 커밋을 사용하라

**하나의 논리적 변경이 완료되면 커밋한다. 사용자가 요청할 때까지 기다리지 않는다.**

- 기준. "이 커밋을 한 문장으로 설명할 수 있는가?" 그렇다면 커밋. 아니라면 아직 변경이 섞인 것이다. 분리하라.
- 좋은 예. "auth 미들웨어 추가". 나쁜 예. "auth 추가하고 UI도 고치고 버그도 수정" (3개로 나눠라).
- 20개의 관련 없는 수정을 쌓아서 개별 롤백 능력을 잃지 마라.
- 의미 없는 커밋은 하지 않는다. 의미 있는 단위만 커밋한다.

참고. 솔로 프로토타입이나 일회성 스크립트라면 속도를 위해 커밋을 느슨하게 묶어도 된다. 핵심은 되돌릴 수 있는 능력이지 형식이 아니다.

---

## 10. 에러는 읽어라. 추측하지 마라

**실제 에러/로그 줄을 읽어라. 기억에서 패턴을 매칭하지 마라.**

무언가 실패했을 때.
- 전체 에러 메시지와 스택 트레이스를 읽어라.
- 내가 예상하는 것이 아니라 실제 로그 출력을 확인하라.
- 원인을 확인하기 전에 "흔한 수정"을 적용하지 마라.
- 불명확하면 print/log를 추가해서 상태를 확인한 뒤 고쳐라.

LLM이 "테스트 실행" 다음으로 가장 자주 건너뛰는 단계다. 에러 키워드로 추측해서 가장 최근 패턴을 적용한다. 그래서 한 줄짜리 버그가 세 파일 리팩토링이 된다.

---

**이 지침이 효과를 발휘하면.** diff에서 불필요한 변경이 줄고, 과복잡으로 인한 재작성이 줄고, 실수 후가 아니라 구현 전에 명확화 질문이 나온다.
