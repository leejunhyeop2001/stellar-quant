# Agent Guidelines

이 문서는 Stellar-Quant 저장소에서 Codex 및 다른 코딩 에이전트가 작업할 때 따라야 할 규칙을 정리한다.

## 1. 코딩 전 확인

- 추측하지 말고 실제 파일과 호출부를 먼저 읽는다.
- 애매한 요구사항은 조용히 해석하지 말고 가정과 선택지를 드러낸다.
- 더 단순한 접근이 있으면 먼저 제안한다.
- 이해가 불분명하면 구현을 멈추고 질문한다.

## 2. 단순함 우선

- 요청받지 않은 기능을 추가하지 않는다.
- 한 번만 쓰는 코드에 추상화를 만들지 않는다.
- 필요 없는 설정화, 확장성, 방어 코드를 넣지 않는다.
- 같은 결과를 훨씬 짧게 만들 수 있으면 줄인다.

## 3. 수술식 변경

- 요청과 직접 관련된 파일과 줄만 수정한다.
- 주변 코드, 주석, 포맷을 이유 없이 개선하지 않는다.
- 기존 스타일이 마음에 들지 않아도 그 스타일에 맞춘다.
- 무관한 죽은 코드를 발견하면 삭제하지 말고 보고한다.
- 내가 만든 변경 때문에 생긴 미사용 import, 변수, 함수는 제거한다.

## 4. 검증 가능한 목표

- 작업 전 성공 기준을 명확히 세운다.
- 버그 수정은 재현 확인 후 수정하고 다시 확인한다.
- 리팩터링은 변경 전후 동작 확인이 가능해야 한다.
- 여러 단계 작업은 간단한 계획과 각 단계의 검증 방법을 먼저 제시한다.

## 5. 편집 전 근거 확보

- `rg` 또는 프로젝트 도구로 관련 구현을 찾는다.
- 편집할 파일과 가까운 호출부를 직접 읽는다.
- 열린 탭, README, 이전 대화 요약은 힌트로만 취급한다.
- 로컬 코드가 가정과 다르면 코드 쪽을 신뢰하고 계획을 수정한다.

## 6. 작업 트리 보호

- 커밋되지 않은 변경은 사용자의 작업물로 간주한다.
- 무관한 변경을 되돌리거나 덮어쓰거나 재포맷하지 않는다.
- 같은 파일에 사용자 변경이 있으면 읽고 그 변경에 맞춰 작업한다.
- 사용자가 명시하지 않은 파괴적 Git 명령은 실행하지 않는다.

## 7. 한국어 출력 규칙

- 사용자가 한국어로 요청하면 한국어로 답한다.
- 한국어 문장을 콜론으로 끝내지 않는다.
- 한국어 문장 끝은 `.`, `?`, `!`를 사용한다.
- 콜론은 코드, 키-값, 시간, 표 라벨 안에서만 사용한다.

## 8. 새 소스 파일 헤더

- 새 소스 파일의 첫 줄에는 파일 역할을 설명하는 한 줄 한국어 주석을 둔다.
- 필수 지시문이나 shebang이 있으면 그 바로 아래에 둔다.
- 설정 파일, 패키지 파일, lock 파일, 생성 파일에는 적용하지 않는다.

## 9. 계획, 체크리스트, 맥락 기록

- 사소하지 않은 작업 전에는 계획을 작성한다.
- 필요한 경우 `checklist.md`에 작업 항목을 체크박스로 정리한다.
- 필요한 경우 `context-notes.md`에 결정과 이유를 계속 기록한다.
- 사용자가 바로 코딩을 요청해도 작업 맥락이 길어질 것 같으면 체크리스트와 맥락 기록 생성 여부를 먼저 확인한다.

## 10. 완료 전 검증

- 코드를 수정했으면 관련 테스트나 빌드, 타입 체크를 실행한다.
- 가장 작은 관련 검사를 먼저 실행하고 위험도가 크면 더 넓게 확인한다.
- 실패하면 실제 에러 로그를 읽고 원인을 확인한 뒤 수정한다.
- 검증을 실행할 수 없으면 이유를 정확히 보고한다.

## 11. 최종 답변의 증거

- 최종 답변에는 실제로 실행한 검증 명령과 결과를 적는다.
- 의도한 검증이 아니라 실제 수행한 검증만 말한다.
- 남은 위험이나 미확인 영역이 있으면 함께 알린다.
- 구체적 확인 없이 완료, 수정됨, 정상 동작 같은 표현을 쓰지 않는다.

## 12. 커밋 단위

- 하나의 논리적 변경은 한 문장으로 설명 가능한 단위로 유지한다.
- 서로 다른 목적의 수정은 섞지 않는다.
- 사용자가 커밋을 원하거나 프로젝트 흐름상 허용된 경우에만 의미 있는 단위로 커밋한다.
- 커밋하지 못하거나 하지 않는 경우에는 변경 내용을 명확히 요약한다.

## 13. 에러 처리

- 실패가 발생하면 실제 에러 메시지와 스택 트레이스를 읽는다.
- 키워드만 보고 흔한 해결책을 추측 적용하지 않는다.
- 원인이 불분명하면 로그나 출력으로 상태를 확인한 뒤 수정한다.

## 적용 기준

이 규칙은 불필요한 변경을 줄이고, 복잡도를 낮추며, 검증 가능한 작업 결과를 남기기 위한 것이다. 사소한 작업에서는 속도와 실용성을 고려하되, 사용자 변경 보호와 검증 보고는 유지한다.
